# noVNC / 整体软件稳定性测试方案 (v3.83)

本文档给出在真机上验证 v3.83 两处修复（`gKeepAliveSec=15` 默认防休眠保活 + 帧存活探针自愈）的可执行方案。
目标：**复现长会话最易触发假死的场景，确认它要么不冻、要么冻了能在 ~10s 内自愈。**

---

## 0. 连接信息 (已确认)

| 项目 | 值 |
|---|---|
| VNC 端口 | `5901` (裸 TCP, 局域网直连可选) |
| noVNC Web 端口 | `5801` |
| Web 客户端 | `http://<设备IP>:5801/vnc.html` |
| WebSocket 代理路径 | `ws://<设备IP>:5801/websockify` |

> 设备 IP 在 App 内「关于 / 网络」或路由器后台查到。PC 与设备需在同一局域网（或设备开热点让 PC 连）。

---

## 1. 手动黑盒测试 (首选, 30 分钟~数小时)

### 场景 A：长时无人值守（原始 Bug 复现）
1. 设备接电源，打开 noVNC 客户端（浏览器 `http://IP:5801/vnc.html`）连上。
2. **把设备放在一边，完全不碰它**，也不要在 PC 端操作。
3. 把 iOS 自动锁定设为 30s（设置→显示与亮度→自动锁定），这样很快进入"易冻"状态。
4. 等待 30~60 分钟，期间定时回来看一眼：
   - ✅ **预期（修复生效）**：设备屏幕保持常亮（保活 ACUnlock 重置了闲置计时），noVNC 画面持续刷新，不冻结。
   - ❌ **失败表现**：设备熄屏 / noVNC 画面定格在某一帧不动 → 说明保活没生效，需查 `gKeepAliveSec` 是否真的为 15、或 App 设置里被关掉。

### 场景 B：强制冻结 → 验证自愈（核心）
1. 保持 noVNC 客户端连接。
2. 在设备上**手动锁屏**（电源键），或把 App 切到后台再回来。这会暂停 `CADisplayLink` → 停止产帧。
3. 静置超过 **60s**（帧存活探针阈值）。
4. 观察：
   - ✅ **预期（修复生效）**：守护进程检测到 60s 无帧 → `exit(0)` → launchd / `trollvncmanager` 监管在 ~10s 内拉起新进程 → 屏幕重新亮起 → noVNC 自动恢复产帧。这是"自愈"。
   - ❌ **失败表现**：进程永久假死、客户端再也连不上 → 探针没触发或监管没拉起，需查日志。

> 注意：场景 B 锁屏是"人为制造冻结"，用来验证自愈链路；真实长会话（场景 A）里因为保活开着，根本不会进冻结，所以两个场景结论要分开看。

### 场景 C：崩溃自愈（监管链路）
1. 连接状态下，用终端（NewTerm / 任意 SSH）在设备上执行：
   `kill -9 $(pgrep trollvncserver)`
2. 观察：~10s 内进程应被监管重启，noVNC 客户端断开后重新连接应当成功。
3. ✅ 恢复 = 监管链（launchd KeepAlive + TRWatchDog）正常。

### 场景 D：抖动压力
- 反复锁屏/解锁、旋转屏幕、切后台/前台、开关飞行模式。
- 每次操作后确认 noVNC 能自动恢复，不出现"要重装才能用"的卡死。

## 2. 自动化监控（客观数据, 推荐跑数小时）

脚本：`tools/vnc_stability_monitor.py`（**零依赖，仅标准库，PC 上需装 Python 3**）。

它连设备 `5801` WebSocket，每 5s 发一次整屏心跳请求，测量：
- 心跳存活 / 帧率 / 内容变化次数
- **冻结检测**：心跳发出后 15s 无响应 → 判假死
- **自愈检测**：连接断开后能否自动重连

### 用法
```bat
cd 工具目录
python vnc_stability_monitor.py <设备IP>
python vnc_stability_monitor.py <设备IP> --ws-port 5801
python vnc_stability_monitor.py <设备IP> --raw 5901      :: 直连裸 VNC 端口
python vnc_stability_monitor.py <设备IP> --heartbeat 5 --freeze-timeout 15
```
- 默认自动写一份 `vnc_monitor_<IP>_<时间>.log` 日志。
- `Ctrl+C` 停止，结束打印汇总（运行时长 / 心跳数 / 冻结次数 / 自愈次数）。

### 怎么判定"稳了"
- 跑 2~4 小时，期间穿插场景 B/C/D 的手动折腾；
- **冻结次数 = 自愈次数**（每次冻都被拉起）→ ✅ 自愈闭环成立；
- 长时静置（场景 A）**冻结次数 = 0** → ✅ 保活生效，根本不进冻结；
- 若冻结次数 > 自愈次数（有冻没拉起）或静置也冻 → ❌ 仍有问题，拿日志来查。

## 3. 其他稳定性维度（不仅是 noVNC）

| 维度 | 验证方法 | 预期 |
|---|---|---|
| Jetsam 内存 | 长跑数小时，用 `vm_stat` / 越狱工具看进程常驻内存是否缓增 | 有 `OhMyJetsam` 豁免，不应被系统杀；若被杀会被监管重启（表现为周期性自愈） |
| 群控 WebSocket | 走 `TVNCHttpServer` 的群控通道（`/api/group/start` 等）长跑 | 不应泄漏连接导致无响应 |
| 无密码鉴权 | 默认无密码；若你在 App 设了 VNC 密码，监控脚本需加 `--password` | 鉴权失败会明确报错，不是假死 |
| 网络瞬断 | 切飞行模式 10s 再恢复 | 客户端断线，恢复后重连成功 |

---

## 4. 出问题时怎么取日志

- 设备端日志在 syslog：Mac 用 `idevicesyslog`（连接设备后过滤 `trollvnc`）；或越狱环境下的 syslog 查看器。
- 监控脚本的 `.log` 文件已含时间戳与冻结/自愈事件，直接发我即可定位。

---

## 结论判读速查
- **静置不冻 + 冻了能活** → v3.83 修复达成，可日常使用。
- **静置也冻** → 保活未生效，查 `gKeepAliveSec` / App 设置。
- **冻了不活** → 探针或监管异常，拿日志排查。
