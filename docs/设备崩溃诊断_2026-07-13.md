# 设备 192.69.0.99 投屏服务崩溃诊断报告（2026-07-13）

> 设备：iPhone SE2（iPhone12,8），iOS 16.0.2（20A380），名 `se2`
> 现象：用户报「服务崩溃了」，8182/5901 均拒绝连接

## 一、本次宕机：直接原因（已恢复）

**时间线**
- 探活：8182/5901 `ConnectionRefused`，22(SSH) `OPEN` → 守护进程没跑，手机在线。
- SSH 进设备（mobile 身份）诊断：
  - `/Library/LaunchDaemons/com.82flex.trollvnc.plist` **不存在**，也没有 `/usr/bin/trollvncserver`。
  - 真机监管链实际是：`TrollVNC.app`（GUI）→ **`trollvncmanager`（监管者）** → `trollvncserver`（守护进程），二进制都在 App 包内。
  - `launchctl list` 里**没有**该 job；`trollvncmanager` 进程也不在（之前是 PID 915）。
  - 守护进程日志最后一行：`03:46 ... GET /api/health` 之后 `Parent process 915 exited` → **监管者 trollvncmanager 在 ~03:46 退出，子进程被一起带走**。
- 因真机**无 launchd plist 兜底**，监管者死后无人重启 → 服务从 ~03:46 静默躺到被发现（约 11 小时）。

**恢复**（已在诊断中执行）：
```
nohup /var/containers/Bundle/Application/9D7D888E-984E-4686-A37A-6A8EB6B60F25/TrollVNC.app/trollvncserver -daemon >/tmp/trollvnc-manual.log 2>&1 &
```
沙箱侧复测：`8182/5901 OPEN`，`/api/status` → `{"status":"running","httpPort":8182,"version":"3.83"}`，`/api/ping` → `{"ok":true}`。**服务已恢复。**

> ⚠️ 注意：手动拉起的守护进程若与 App 之后拉起的 `trollvncmanager` 抢 5901 端口会冲突；冲突时 `kill` 掉重复进程再重拉即可。

## 二、深层根因：守护进程周期性 SIGABRT 硬崩（这才是「服务会挂」的真凶）

- 真机 `trollvncserver` 崩溃报告**共 25 份**（最早 7/9，每天都有），即**周期性硬崩溃**。
- 最新一份（`trollvncserver-2026-07-12-214216.ips`）签名：
  - `exception: {type:"EXC_CRASH", signal:"SIGABRT"}`
  - `asi: {"libsystem_c.dylib":["abort() called"]}`
  - 线程6 `__abort` / `abort`；线程1 `_dispatch_call_block_and_release`；主线程卡在 `mach_msg`/CFRunLoop
  - → **后台 GCD 异步块里直接 `abort()`**（断言失败 / 未捕获异常导致整个进程自杀）
- **与 v3.83 修复的区别**：
  - Fix 2 帧存活 watchdog 用 `exit(0)`（软退出，交 manager 重启）→ 只兜「假死/不产帧」。
  - 这些崩溃是 `abort()`（**硬崩**），watchdog 兜不住。
  - 平时靠 `trollvncmanager` 重启掩盖，用户无感；直到本次 manager 自己先挂，才彻底暴露。
- 崩溃时间横跨 7/9–7/12，**早于本次（7/12）测试**，故非测试引入。
- **精准定位需符号化**：用匹配版本的 dSYM（`build_output/dsym-*`）对崩溃偏移 symbolize，才能锁到具体源文件/行。

## 三、结论

| 问题 | 性质 | 状态 |
|------|------|------|
| 本次服务宕机 | 监管者 `trollvncmanager` 退出 + 无 launchd 兜底 | ✅ 已手动拉起恢复 |
| 周期性 SIGABRT 硬崩 | 后台异步任务 `abort()`，真实稳定性缺陷 | ⏳ 待源码修复（需 dSYM 符号化定位） |

**一句话**：你最早说的「iOS 16 长时投屏服务会挂」是真存在的，但主因不是我们修的「假死」，而是**守护进程自身周期性 abort 崩溃**——平时被监管者重启盖住了，这次监管者先挂才暴露。

## 四、后续可选项（等你定）

1. **符号化定位 SIGABRT**：用匹配 dSYM 解析崩溃偏移 → 锁到具体源文件/行 → 修 bug（治本）。
2. **装 launchd plist 自动拉起**（治标）：需 **root SSH**（你只给了 mobile，写不了 `/Library/LaunchDaemons`）。部署指向 App 包内二进制的 plist + `launchctl load`，崩溃即自动重启。
3. **关保活重测 T2**（之前约定的 Fix2 验证）：仍需 root 改 plist 或你配合；与本次事故无关，建议先稳定观察。
