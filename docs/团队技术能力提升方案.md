## 1. 现状体检（事实依据）

**已有的好习惯（值得保留并强化）：**
- 有 `.clang-format`（LLVM 风格，120 列）——代码风格有基线。
- 有 GitHub Actions CI：4 套 scheme（default / rootless / roothide / bootstrap）矩阵构建，并能自动打 tag 发布。
- CHANGELOG 维护较认真，记录了 iOS 13 兼容性、LaunchDaemon 等真实修复。

**明显的短板（客观事实）：**
- ❌ 没有 `.clang-tidy`——**零静态分析**。
- ❌ 没有任何测试——**0 个单元/UI 测试**。
- ⚠️ CI 只"构建"，不跑 lint、不跑 clang-tidy、不跑测试，等于没有质量门禁。
- ⚠️ `Makefile` 中 `GO_EASY_ON_ME := 1`，且 CFLAGS 含 `-Wno-unknown-warning-option`、`-Wno-unused-but-set-variable`——**主动压制编译器警告**。
- ⚠️ 版本号在三处不一致（详见第 2 节）。

## 2. 核心差距（按严重度分级）

### 🔴 高（直接影响稳定性 / 可维护性）

1. **零测试覆盖**
   网络、进程拉起（`TVNCServiceCoordinator`）、私有 API 调用均无测试。任何重构都是"开盲盒"。这是团队能力最大的短板。

2. **崩溃处理器异步信号不安全** —— `app/MatisuXCS/MatisuXCS.m:7-39`
   用 `signal()` 注册，handler 内调用 `NSDateFormatter` / `NSString` / `writeToFile:`（**均非异步信号安全**）。崩溃瞬间可能二次崩溃或死锁。
   修复：改用 `sigaction` + 仅用 `write()` 系统调用（见《重构示范》），或接入 PLCrashReporter / KSCrash。

3. **版本号三处不一致**
   - `Makefile:1` → `PACKAGE_VERSION := 3.53`
   - `CHANGELOG.md` 最新条目 → `3.44`（落后于实际构建）
   - `prefs/TrollVNCPrefs/TVNCDeviceInfoController.m:23` → 兜底 `#define PACKAGE_VERSION "3.5"`（过时）
   CHANGELOG 与发布脱节，兜底值过时，发布与故障排查会混乱。

### 🟡 中（健壮性 / 安全 / 可维护性）

4. **生产代码大量裸 NSLog** —— `app/MatisuXCS/MatisuXCS.m` 共 18 处
   且打印了 bundle id、URL scheme、错误明细，存在信息泄露风险，且无日志级别。需统一日志宏，release 关闭 debug。

5. **私有 SPI 前向声明无可用性保护** —— `app/TrollVNC/TrollVNC/TVNCServiceCoordinator.m:31-35`
   `SBSLaunchApplicationWithIdentifierAndURLAndLaunchOptions` 等直接 `FOUNDATION_EXPORT` 声明并调用，跨 iOS 版本易崩。团队在 CHANGELOG 已做过大量 iOS 13 fallback，说明有意识，但 Coordinator 这一处缺保护。

6. **`openSettingsAndExit` 控制流复杂且含竞态** —— `app/MatisuXCS/MatisuXCS.m:141-239`
   嵌套 `dispatch_after`、多处 `exit(0)`；`openURL:options:completionHandler:` 后仅延迟 0.5s 即假设成功并退出，completionHandler 可能异步返回失败 → 用户未被跳转却被杀进程。

7. **`dlopen` 后未 `dlclose`、私有 API 调用无返回检查**（多处）
   句柄泄漏；`objc_msgSend` 强转调用后未判空、未检查返回值。

### 🟢 低（优化项）

8. **轮询式健康检测** —— `app/TrollVNC/TrollVNC/TVNCServiceCoordinator.m:98-106`
   每 3s `NSTimer` 轮询 socket；短期可接受，但应有 backoff 与更优通知机制。

9. **单文件职责过重** —— `app/MatisuXCS/MatisuXCS.m`
   把"打开设置"全部逻辑压在一个 ~240 行方法，违反单一职责，后续难维护。

## 3. 分阶段提升路线图

### 阶段一：护栏先立（0–4 周）
- 引入 `.clang-tidy`，开启 `-Wall -Wextra`，逐步清理 `GO_EASY_ON_ME` 与 `-Wno-*` 抑制项，消灭警告。
- CI 增加 **lint 门禁**（`clang-format --dry-run`）+ **clang-tidy 检查**，PR 不通过不准合。
- 统一日志宏 `TVNCLog`（error / warn / info / debug），release 关 debug，替换散落 NSLog。
- 修复崩溃处理器（见《重构示范》）+ **版本号单一来源**（脚本从 Makefile 生成 VERSION，CHANGELOG 强制同步）。

### 阶段二：测试与文化（1–3 月）
- 搭测试脚手架：逻辑层用 XCTest；进程/网络用 mock + 集成测试。优先给 `TVNCServiceCoordinator`、`TVNCHotspotManager` 写单测。
- 建立 **PR 模板 + 代码评审 Checklist**（见附件），要求至少 1 人评审 + checklist 全过。
- 私有 API 调用统一收敛到 `SPIWrapper` 层，全部带可用性检查与兜底。
- 引入 Issue/PR 模板、语义化版本（SemVer），CHANGELOG 由工具生成（如 `standard-version`）。

### 阶段三：质量内建（3–6 月）
- 覆盖率门禁（核心模块 ≥ 60%），CI 卡覆盖率。
- 关键路径接入崩溃上报（PLCrashReporter），建立崩溃看板。
- 季度代码健康度复盘（技术债、复杂度、覆盖率趋势）。
- 团队内部分享 / 结对编程，沉淀《私有 API 兼容手册》《越狱工具健壮性指南》等内部知识库。

## 4. 工程规范要点（立即执行）

- **单一版本来源**；CHANGELOG 与发布强一致。
- 所有私有 API 走封装层 + 版本保护。
- 日志分级，禁止裸 NSLog 进主干。
- 崩溃回调只用异步信号安全函数。
- 每个 PR 附测试，或明确说明为何不可测。
